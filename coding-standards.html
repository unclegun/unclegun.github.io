Web Coding Standards and Processes

Overview: This document captures proposed standards and processes to improve development efficiency, sustainability, and team alignment. It is organized as a brainstorming foundation and requires team review and feedback.

Overarching Goals

Foundational principles that guide all subsequent decisions:

  • Efficient development: Reduce cycle time, minimize re-work, and enable rapid iteration.
  • Sustainability: Avoid "foreign land" minefields (hidden technical debt); prevent bolt-on change requests.
  • Smart customizability: Total stakeholder consideration; flexible without chaos.
  • Smart stovepipes: Clear boundaries between modules; minimize coupling.
  • AI-generatable: Codify patterns so tools and teams can generate conformant code.
  • Increased quantifiability: Measurable estimates; eased effort planning.
  • "Juice worth the squeeze": Value delivered must justify the effort invested.
Intent: These goals frame every downstream standard. If a standard doesn't serve at least one goal, question it.

Front-End

site.css Conventions (Component-Level Organization)

CSS should be organized by component, not by property. Group related elements together:

  • Form elements: inputs, selects, textareas, labels, validation states.
  • Cards: container, header, body, footer, shadow, border states.
  • Messages: alerts, toasts, banners; success, info, warning, error variants.
  • Buttons: primary, secondary, outlined, disabled, loading states.
  • Headers: page title, breadcrumbs, record identity, status chips.
  • Navigation: sidebar, tabs, breadcrumbs, pagination.

Forms & UX Patterns

  • Field layout: Label above input; consistent spacing and alignment.
  • Validation: Client-side hints; server validation on submit; clear error messages.
  • Edit mode options:
    Choose wisely (expand to view)
    • Modal: For small, focused edits; interrupts flow; good for confirmations.
    • Inline partial: For quick changes within context; minimal page reload.
    • Separate page: For complex, multi-step edits; auditable journey.
Open Questions

Success/Info/Error Messages

Consistent messaging improves UX and reduces support burden.

  • Success: Brief, celebratory; action successful.
  • Info: Educational; helps user understand context.
  • Error: Specific, actionable; tell user how to fix.
  • Warning: Caution or alert; user should read before proceeding.

DataTables.js & Site Navigation

  • Column headers are clickable for sort; indicate current sort direction.
  • Search is full-text; filters are additive and isolated.
  • Row selection for bulk actions (export, update, delete).
  • Pagination keeps performance snappy; default to 10–20 rows per page.

Data Tier

Data Access Layer (DAL) Rules

Standard Intent: All SQL happens in the DAL. This centralizes database logic, eases testing, and simplifies auditing.
Model options
  • Models for every entity? Ensures consistency; adds boilerplate. More maintainable long-term.
  • Models for major entities only? Faster to ship initially; risk of drift and inconsistency.
  • MVC-style models for views? Recommended: Core entities always have models; view-specific models as needed for presentation.
Where does SQL live?
  • All SQL centralized in DAL? Yes. Easier to audit, test, version, and refactor.
  • Ad-hoc calls in PageModel acceptable? No. Hidden dependencies; harder to test; complicates auditing.
Full-entity vs partial updates?
  • Full-entity updates: Simpler logic; higher versioning (RowVersion) overhead.
  • Partial/ad-hoc: More control; risk of orphaned fields.
  • Recommendation: DAL provides full-entity methods by default. Partial updates when justified (e.g., high-frequency field changes).
Open Questions

Business Logic Placement

Where does business logic live?
  • DAL strictly for DB calls? Clean separation; but business rules end up scattered across PageModel and services.
  • DAL houses business logic for entity? Recommended: Entity-specific validation, calculations, and state transitions belong in DAL or a domain service (shared, testable).

Database Helpers & Advanced SQL Server Features

  • DB call efficiency: Minimize round-trips; use batch operations for bulk work.
  • EF guidance: Use EF for CRUD; stored procs for complex aggregations.
  • SQL Views: Denormalize read-heavy queries; keep views indexed and tested.
  • Stored procs: For complex multi-step operations, data migrations, or bulk updates. Document purpose and usage.
  • Functions & Triggers: Use judiciously; document intent, side effects, and test thoroughly.

Data Modeling

Normalization

Organize schema to minimize redundancy and enforce consistency:

  • First Normal Form (1NF): No repeating groups.
  • Second Normal Form (2NF): No partial dependencies.
  • Third Normal Form (3NF): No transitive dependencies.
  • Document exceptions: Justified denormalization with clear rationale (e.g., performance).

Audit Logging

Standard Intent: Track who changed what, when, and why. This is critical for compliance, debugging, and accountability.
  • Schema: Audit table per major entity or centralized audit log.
  • Implementation: Use triggers or DAL logic; document chosen approach.

Lookup / Enum / Constants / Bits

Manage reference data consistently:

  • Lookup tables: For mutable, user-managed values (e.g., department, location, team).
  • Enums / bits: For immutable, small sets (e.g., status, permission flags, role).
  • Code constants: Map enum values to user-friendly display names in a single, centralized place.
  • Documentation: Maintain a data dictionary; explain business meaning of each value.

Web Code

Global(-ish) Variables

Centralize configuration and reduce hidden dependencies:

  • appsettings: Connection strings, feature flags; environment-specific.
  • tblGlobalVariables: Runtime-configurable settings; no code deploy needed for operational changes.
  • Object-level: Singleton or DI-injected services; immutable after initialization.

Code Comments — Enforceable Standards

Standard Intent: Comments should explain intent, not restate code.
  • Function definitions: Summarize purpose, key inputs, outputs, exceptions, and any non-obvious behavior.
  • Versioning notes: Mark rework, breaking changes, deprecations with date and owner.
  • CR tracking: Reference change request ID or ticket; link to decision.
  • Why over what: Code shows what; comments explain why (business logic, workarounds, gotchas).

Enforceable Guidelines for Code Repetition

Standard Intent: DRY (Don't Repeat Yourself) is valuable, but copy-paste to avoid tight coupling is sometimes justified.
  • If two modules copy-paste the same snippet, that's probably a shared utility candidate.
  • If one module copy-pastes code from another and documents which version it copied from, that relationship is explicit and maintainable.
  • Never hide shared code behind complex inheritance or dynamic dispatch that obscures intent.
  • Document the exception in code comments: why is this duplicated? When should it be consolidated?

Template & Solution

Utilities (Expansion & Usage)

Share common patterns across teams via utility libraries:

  • String formatting, date/time helpers, crypto utilities.
  • Data validation, masking, normalization functions.
  • Logging, error handling, retry logic.
  • Versioning: Semantic versioning; document breaking changes; notify teams of updates.

Third-Party Tools

In-template tools
  • DataTables.js: Table search, sort, filter, paginate.
  • Bootstrap: Grid, components, utilities.
  • HTMX: Partial page updates without React/Vue; keep it simple.
  • Chart.js: Client-side visualization; data from server JSON.
Outside-template on APL
  • Swagger/OpenAPI: API documentation and client generation.
  • Entity Framework: ORM for C# data access.
  • NUnit / xUnit: Unit testing framework.
  • Azure DevOps: CI/CD, Agile boards, version control.

Misc

Mock-ups vs Early Drafts

Clarify expectations before coding begins:

  • Wireframes: Low-fi, structure and flow; quick iteration; rapid feedback.
  • Mock-ups: High-fi visuals; more detail; slower to produce; good for stakeholder alignment.
  • Early drafts: Minimal requirements to check-in; document gaps and assumptions explicitly.

Data Migration & Revisit Coding Standards

  • Migrations should be repeatable and safe to run multiple times.
  • build staging plan on dev; finalize implementation by UAT

Lean Development & Customer Expectation Management

  • Deliver in small, testable increments; gather feedback early and often.
  • Define MVP upfront; communicate what is not included in each release.
  • Use roadmaps to show future work and manage expectations.
  • "Done" means tested, documented, and deployable—not just coded.

Team Requirements & Mindset

  • Champion: Executive sponsor who unblocks decisions and communicates importance.
  • Subteams: Functional groups (Back-End, Front-End, QA, Security) with clear responsibilities.
  • Review counsel: Senior engineers and architects; private governance channel for contentious decisions.
  • Cultural note: "Standards are great… as long as they are mine." Involve teams in standard-setting to earn buy-in and ownership.

Alternative Frameworks

These standards assume a hybrid Agile/waterfall approach. Consider:

  • Kanban: Continuous flow; good for ops and maintenance.
  • Scrum + SAFe: Scaled agile; suitable for large programs with dependencies.
  • Lean + CI/CD: Fast feedback loops; requires high engineering maturity and strong automation.
